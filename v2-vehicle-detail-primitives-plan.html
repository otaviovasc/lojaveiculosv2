ID hygiene rule

UI must prefer plate, stock number, public slug, vehicle title, customer name, store name, or a human status. Raw database IDs, UUIDs, unit IDs, tenant IDs, and listing IDs stay inside state, routes, API payloads, tests, and logs only.

Backend and admin vehicle detail roadmap

Publish readiness model

Build a shared readiness helper and backend-backed publish result that explains exactly why a vehicle is public or blocked. Public exposure should require published listing status, public slug, storefront visibility, non-deleted listing, visible unit/media, and published store settings.

Turn anuncio and vitrine tabs into real public listing controls

Replace local mock state with existing listing status, visibility, public slug, theme/preview, and storefront API data. The admin should show the same availability that the storefront repository will enforce.

Finish documents and RENAVE surfaces

Reuse the existing document upload/list APIs and media workspace patterns instead of adding another local document shell. Keep fake RENAVE/checklist cards clearly unavailable until backed.

Polish reserve, sell, and release workflow UX

The backend workflow endpoints already perform the canonical status changes and return refreshed detail DTOs. Next UI work should add better success states, generated document links, finance feedback, and backend error messages.

Public exposure QA loop

For every publish/unpublish flow, verify admin readiness, public listings, public detail, media grouping, lead submission, mobile layout, and the negative case where the listing must disappear.

Quick wins

  • Promote vehicle detail metric/readiness rows into shared primitives.
  • Replace remaining technical placeholders with business copy.
  • Add a reusable selected-state recipe for gallery, block list, and tab controls.
  • Apply one empty/error/loading state family across builder and admin modules.
  • Run browser screenshots for admin detail, storefront detail, and builder blocks after every UI slice.

Do not change casually

  • Do not replace v2 UI with Centro Imovel UI wholesale.
  • Do not rewrite backend architecture for publish/workflow; extend existing services.
  • Do not expose tenant, store, listing, unit, lead, or user IDs in normal UI.
  • Do not add duplicate panel/card/form primitives when existing ones can be extended.
  • Do not make public tenant pages dark theme.
