# Feature Inventory

> This is a technical surface inventory, not a statement that a feature is
> commercially safe, adopted by customers, compliant, supportable, or ready for
> V1 cutover. Commercial launch is gated separately below.

Every migrated feature needs one row before implementation. Status values:

- `Production-ready`: runtime route/UI exists, real adapters are wired, and the
  remaining setup is environment/provider configuration.
- `Runtime-ready`: route/UI exists and production adapters are wired for owned
  data, but one external provider is not enabled until credentials/endpoints are
  supplied.
- `Placeholder`: visible navigation exists, but the feature is intentionally not
  implemented yet.
- `Migration-deferred`: product surface exists, but historical V1 migration is
  intentionally skipped for now.
- `Critical gap`: the capability blocks safe commercialization, retention, or
  migration and must be resolved before adding speculative product breadth.

## Commercial Launch Gate

A route, screen, adapter, or provider-readiness check is not enough to clear a
commercial launch. A feature is customer-ready only when all applicable items
below are evidenced:

1. At least one real design-partner store completed the workflow with its own
   data and accepted the outcome.
2. The workflow has an explicit activation or adoption metric and an internal
   owner can see failures without querying raw production data.
3. Billing, entitlement, quota, provider cost, cancellation, and support
   ownership are defined.
4. Tenant isolation, private-data storage, secrets, audit, idempotency, retry,
   and rollback behavior are proven for the risk class.
5. Required external contracts and Brazilian regulatory fields are confirmed
   against current primary sources.
6. The V1-to-V2 migration and rollback path exists when an active customer
   depends on equivalent V1 behavior.

Current P0 commercialization blockers are maintained in
[`strategy/product-operating-model.md`](strategy/product-operating-model.md).
