Current locks

No-Regret Decisions

  • FIPE is cached in V2 tables; UI dropdowns never call FIPE directly.
  • Vehicle reserved status is a workflow, not a bare status flip.
  • Sales draft reserve/close must reuse the canonical vehicle workflow finalizer; they must not create duplicate sale/payment rows or parallel finance/document logic.
  • Vehicle lifecycle transitions (unit reserved/sold/released and listing publication/status updates) must include actor-linked logs and before/after diff capture.
  • Reserved units must always generate a reservation receipt document before status lock.
  • Sold state must emit exactly 4 sale document kinds: sale_contract, sale_receipt, delivery_term, power_of_attorney.
  • Finance Slice 1 owns manual and recurring entries, pay/cancel workflows, vehicle-linked expense views, dashboard KPIs, commission rules, and R2-backed evidence.
  • Finance mutations must audit actor, permission, reason, before/after values, linked entity ids, recurrence id, payment/cancel state, commission rule id, and attachment references.
  • CRM WhatsApp migrated paths are V2-owned. Repasses remains a behavior reference and later import/backfill source, not a runtime source of truth for the migrated ZAPI inbox, webhooks, sends, tags, templates, or assignment flows.
  • Audit database is separate; product tables do not depend on audit tables.
  • Railway is the primary deploy target for V2 services/databases.

North Star

Build V2 as the durable main product surface: Vite + React frontend, scoped backend domains and feature controllers, strong typing, auditable services, enforceable permissions, and a migration path from current Loja V1 data without feature degradation.

Current WILL: V2 is the canonical architecture surface and contracts must be contract-driven and reusable across frontend/backend. V2 will not keep scaffold responses or workaround behavior as user-visible behavior, and it will not preserve compatibility-shaped contracts from legacy systems. Canonical V2 DTOs must drive both API and UI contracts for inventory, FIPE-backed vehicle catalog, media gallery, and documents. Vehicle listing + unit split is the foundation, not an optional refactor.

The architecture is now explicitly lifecycle-first: listing creation and publication are listing-owned; reservation, release, sale, delivery, and post-sale doc/ledger completion are unit-owned workflows in the canonical state machine for V2.

Vehicle as foundation

Everything in V2 is built around vehicle_listings + vehicle_units and their lifecycle, not around legacy mixed objects. This keeps inventory, costs, documents, media, and sales accounting on one consistent graph.

Stage 1 Discover

Catalog-backed listing creation with FIPE selection, unit media upload, and complete public/admin payloads.

Stage 2 Publish

Visibility/status transitions become persisted entity events with actor + before/after deltas.

Stage 3 Reserve

Reservation is a transactional workflow: store user action, selected vehicle unit, customer snapshot, signal, document, unit status update, and payment draft.

Stage 4 Sell

Sale creation closes the selected unit, creates sale documents, generates financial links, and marks the listing sold-out only when no open units remain.

Stage 5 Close

Post-sale handoff records delivery, settlement, tax/doc links, and immutable audit evidence.

Vehicle Backend Audit

Audit date: 2026-06-25. Current V2 backend uses vehicle_listings as the canonical commercial listing and vehicle_units as the canonical physical/stock item. They work together: listings own public/admin presentation, pricing, catalog snapshot, visibility, and search; units own plate, VIN, stock number, color, acquisition state, unit costs, unit media, reservation/sale locking, and the sold unit reference.

Implemented vehicle tables

vehicle_listings, vehicle_units, vehicle_media, vehicle_plate_lookups, vehicle_costs, vehicle_price_history, vehicle_status_history, FIPE catalog tables, lead_vehicle_interests, and vehicle_provider_listings all have backend schema and at least one runtime read/write path.

Shared document model

There is no standalone vehicle_documents, vehicle_document_kinds, or sale_documents table in V2. Vehicle and sale documents are stored in documents and connected by document_links.target_type values such as vehicle_unit, sale, and sale_payment.

Backend hardening completed

Broad listing edits now record price/status history, reserve/sell now guard invalid listing + unit states, reservation writes unit status history only, sale writes unit status history and listing sold_out history only when stock is exhausted, sales draft reserve/close reuse the same workflow finalizer inside the sales transaction runner, and workflow validation/state errors are mapped to client 400/409 responses instead of generic 500s.

Remaining vehicle backend gaps

Reservation auto-expiry scheduling still needs a policy; document replace/remove plus CDN invalidation still need final backend governance; assistant-safe inventory tools still need an audited backend contract.

Known listing/unit logic debt

Unit reserve/sell/release, costs, checklists, acquisition, documents, and media are now unit-primary. UI/status filters must continue treating listing status as publication state and unit status as stock lifecycle state.

Intranet Direction

Same-domain routing can work across providers through DNS and public HTTPS. Same-intranet/private service communication is different: Render private networking applies to services in the same Render workspace and region, while Railway private networking applies inside a Railway project environment. If private service-to-service traffic is a hard V2 requirement, the recommended target is consolidating runtime services onto one provider or adding a deliberate cross-provider private networking layer.
